## Architecture

### Client Hierarchy

The SDK uses a hierarchical client structure:

```
MyOrganization (top-level client)
├── OrganizationDetails (organizationdetails.Client)
└── Organization (organization/client.Client)
    ├── Configuration
    │   ├── APIs
    │   └── IdentityProviders
    ├── Domains
    │   ├── IdentityProviders
    │   └── Verify
    ├── IdentityProviders
    │   ├── Domains
    │   └── Provisioning
    │       └── ScimTokens
    ├── ClientGrants
    ├── Clients
    ├── Invitations
    └── Members
```

### Core Packages

- **client/**: Top-level client (`MyOrganization`) that aggregates all API resources
- **organization/**: Organization-related API resources organized by domain
- **core/**: Core SDK functionality
  - `api_error.go`: Structured error types compatible with `errors.Is` and `errors.As`
  - `http.go`: HTTP transport configuration
  - `request_option.go`: Request option types
- **internal/**: Internal utilities not meant for external use
  - `caller.go`: HTTP request execution with retry logic
  - `retrier.go`: Exponential backoff retry mechanism
  - `error_decoder.go`: Response error parsing
  - `query.go`: URL query parameter encoding
  - `explicit_fields.go`: Explicit null value handling via bitmasks
- **option/**: Public request options (`WithToken`, `WithBaseURL`, `WithHTTPClient`, etc.)

### Request Options Pattern

The SDK uses a functional options pattern for configuration:

```go
// Client-level options (applied to all requests)
client := client.newWithOptions(
    option.WithToken("<token>"),
    option.WithHTTPClient(&http.Client{Timeout: 5 * time.Second}),
    option.WithMaxAttempts(3),
)

// Request-level options (applied to single request)
response, err := client.Organization.Domains.Create(
    ctx,
    request,
    option.WithToken("<override-token>"),
)
```

### Error Handling

All API errors are returned as `*core.APIError`:

```go
response, err := client.Organization.Domains.Create(...)
if err != nil {
    var apiError *core.APIError
    if errors.As(err, &apiError) {
        // Access apiError.StatusCode, apiError.Body, etc.
    }
}
```

### Retry Behavior

- Automatically retries requests on 408, 429, and 5XX status codes
- Uses exponential backoff (configurable via `option.WithMaxAttempts`)
- Respects `Retry-After` header when present
- Default: 2 retry attempts

### Explicit Null Values

Use setter methods to send explicit `null` in JSON:

```go
request := &ExampleRequest{}
request.SetName(nil)  // Sends {"name": null} instead of omitting the field
```

This works by flipping bits in an internal `explicitFields` bitmask.

## Testing

Tests use WireMock for HTTP mocking. Test fixtures are stored in:
- `wiremock/wiremock-mappings.json`: Request/response mocks
- `wiremock/docker-compose.test.yml`: Docker Compose configuration

Each API resource has a corresponding test package (e.g., `organization/domains/organization_domains_test/`).
